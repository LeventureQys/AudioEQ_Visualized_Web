## Agent开发规范

以下内容只需要Agent看就行了，如果你是人类你可以不看。

### 适用规模说明
> **本范式仅适用于大规模 Version**（需求点数量 ≥ 5，且存在明显可并行的子任务，或代码依赖复杂、需要多个 SubAgent 并行开发）。  
> **对于小规模 Version**（需求点少、依赖简单、预估工作量小），只需由 **单个 MainAgent 独自完成所有开发与测试**，无需拆分 Stage、SubAgent 等复杂流程。  
> 下文描述的内容均为大规模 Version 下的规范。


### 1. 核心思想

#### 1.1 基础思想（Version-Stage 拆分与并行）
- 每个 **Version** 承载一组大的需求，使用语义化版本号（如 v1.2.0）。
- 将 Version 拆分为多个 **Stage**，Stage 之间按**代码依赖关系**（或业务顺序）串行执行。
- 每个 Stage 内部拆分为多个 **SubStage**，SubStage 之间**完全并行**，由多个 SubAgent 并发开发。
- SubAgent **之间不允许直接沟通**，所有协调必须通过 MainAgent 提前规划。
- 完成每个 Stage 后执行**增量系统测试**（只测新增/影响的功能），Version 全部完成后执行**全量系统回测**。
- 失败时重试 SubStage（最多3次），若仍失败则由 MainAgent 重新规划 Stage。

#### 1.2 核心思想一：事前零猜测（问题清单驱动）
> **在正式开始每一个 Version 的开发之前，必须向开发者（用户）澄清 Version 中的每一个细节。绝对禁止任何猜测或带着不确定性自行开发。**  
> 所有不确定的问题，在对应版本号或对应 Stage 文件夹下新建一个 `问题清单.md`，逐条写入。问题清单可能经过多轮问答，直到所有问题都得到明确答复后，才允许 Agent 开始真正的开发。

#### 1.3 核心思想二：事中/事后不确定性追加
> **完成一个 Version 或 Stage 的过程中，如果 MainAgent 或 SubAgent 遇到任何不确定性（例如临时决定、留空、假设），也必须将这些不确定性追加到 `问题清单.md` 的末尾。**  
> 必须请用户多轮解答完这些新问题之后，再进行一轮补充开发和自测，直到 Agent 不再有任何疑问为止。

### 2. 角色与职责

| 角色 | 职责 |
|------|------|
| **MainAgent** | 1. 拆分 Version 为 Stage（按代码依赖顺序）<br>2. 将每个 Stage 拆分为可并行的 SubStage<br>3. 分发 SubStage 给 SubAgent<br>4. 协调测试、汇总结果、处理重试与重新规划<br>5. 管理 Stage 文档、测试报告以及 **问题清单**<br>6. **自行审核 Stage 设计文档**（无需其他角色）<br>7. **确保所有不确定性在开发前澄清，开发中产生的新不确定性及时追加到问题清单并寻求用户解答** |
| **SubAgent** | 1. 接收已拆分好的 SubStage 任务<br>2. 直接开发（不自行拆分任务）<br>3. 完成开发后自测，并报告给 MainAgent<br>4. 若失败，配合重试或重新分配<br>5. **禁止与其他 SubAgent 直接沟通**；如有需要共享的信息，由 MainAgent 在设计阶段明确提供<br>6. **开发中遇到任何不确定性（包括设计空白、依赖不明确等），立即停止并报告 MainAgent，由 MainAgent 追加到问题清单** |

### 3. 开发流程

#### 3.1 Version 规划（含问题清单）

1. **确定大需求组**（例如“订单系统性能优化”），分配语义化版本号（如 v2.1.0）。
2. **创建 Version 级问题清单**：  
   - 在 Version 文件夹下新建 `问题清单.md`。  
   - MainAgent 详细阅读需求，列出所有不确定的点：接口定义不清？依赖组件版本？性能指标？边界条件？  
   - 将问题清单提交给用户，进行多轮问答，直到所有问题得到明确答案。  
   - **在问题清单全部关闭之前，不允许进入下一步拆分**。
3. **拆分为小需求点**：  
   - 基于澄清后的需求，优先按**代码依赖关系**解耦。  
   - 如果无法完全解耦，允许保留必要依赖，此时该依赖关系决定 Stage 的串行顺序。  
   - 也可以按业务顺序划分，但必须保证前后 Stage 之间的接口稳定。
4. **排列 Stage 顺序**：  
   - 将所有小需求点按依赖关系排序，形成线性序列。  
   - 将该序列切分成若干个连续的 **Stage**（Stage 之间串行执行）。

#### 3.2 Stage 内部规划（含子问题清单）

每个 Stage 开始前，必须重复问题清单流程：
1. **创建 Stage 级问题清单**（或追加到 Version 主问题清单中，建议独立）：  
   - 在对应 Stage 文件夹下新建 `问题清单.md`。  
   - MainAgent 针对该 Stage 的具体任务，列出所有需要澄清的细节：输入输出格式、跨 SubStage 契约、异常处理策略等。  
   - 提交用户解答，全部关闭后才允许拆分 SubStage。
2. **拆分 SubStage**：  
   - 将一个 Stage 包含的小需求点继续拆分为**无相互依赖**的 **SubStage**（最小工作单元）。  
   - SubStage 之间可完全并行。  
   - 一个 SubAgent 可以被分配多个 SubStage（顺序或并发执行由 SubAgent 自己控制）。  
   - **关键约束**：设计 SubStage 时必须假设 SubAgent 之间无法互相通信。所有需要跨 SubStage 共享的数据、接口、约定，必须由 MainAgent 在 Stage 设计文档中明确写出。
   - **文档自足性约束**：Stage 设计文档（以及每个 SubStage 的任务书）必须满足**"零上下文可开发"标准**（见第 6.1 节）——一个全新 session 仅凭文档即可动手，无需依赖任何口头约定或本次对话历史。

**示例**：  
Stage 2 包含需求点 2.a、2.b、2.c，三者无代码依赖，但 2.c 需要用到 2.a 输出的某个数据结构。  
→ MainAgent 必须在设计文档中明确规定该数据结构的格式、存储位置，并分发给 2.a 和 2.c 的 SubAgent。

#### 3.3 执行流程（单 Stage，含开发中问题追加）

1. **MainAgent 拆分当前 Stage 为 SubStage**（明确每个 SubStage 的输入、输出、验收标准，以及跨 SubStage 的契约）。
2. **MainAgent 审核并确认 Stage 设计文档**（自行审核，通过后开始分发）。
3. **分发 SubStage 给空闲 SubAgent**。
4. **SubAgent 开发**：  
   - 完成代码实现。  
   - **开发中若遇到任何不确定性问题**（例如文档未说明的边界、依赖行为未定义、需要临时做决定等），**SubAgent 立即停止并报告 MainAgent**。  
   - MainAgent 将问题追加到当前 Stage 的 `问题清单.md` 末尾，并提交给用户解答。  
   - 在用户解答之前，该 SubAgent 暂停该任务，可切换其他已澄清的子任务。  
   - 用户解答后，MainAgent 将答案反馈给 SubAgent，继续开发。  
   - 本地单元测试通过后，向 MainAgent 报告完成。
5. **MainAgent 等待所有 SubStage 完成**（期间持续处理追加的问题）。
6. **MainAgent 在共享测试环境触发测试**：  
   - **全量单元测试**（所有已有单元测试）。  
   - **增量系统测试**：只测当前 Stage 新增或影响的功能（使用冒烟测试 + 关联回归用例）。
7. **根据测试结果决定**：  
   - ✅ 全部通过 → 该 Stage 完成，进入下一 Stage。  
   - ❌ 有失败 → 定位到具体的 SubStage，通知对应 SubAgent 重试（仅重试该 SubStage）。  
     - 重试次数 ≤ 3 次。  
     - 若 3 次后仍失败，MainAgent 停止当前 Stage，**重新分析任务规划**，可能：
       - 调整 SubStage 拆分方式。
       - 修改该 Stage 的依赖顺序。
       - 甚至重写整个 Stage 的分配方案。
8. **Stage 完成前交付物**：  
   - 设计/实现文档（说明该 Stage 完成的内容、接口变化、跨 SubStage 契约等）。  
   - 测试报告（单元测试 + 增量系统测试结果）。  
   - **最终的问题清单**（包含所有已解答的问题，以及开发中追加并解答的问题）。  
   - 三者由 MainAgent 收集存档。  
   - **Stage 不打 tag**，仅记录在内部文档中。

#### 3.4 Version 完成后的收尾

- 所有 Stage 都完成后，执行一次**完整的全量系统测试**（覆盖整个系统，包括所有历史功能和本次 Version 全部新功能）。
- **检查整个 Version 的问题清单**：确保所有问题（Version 级 + 各 Stage 级）都已关闭。如果还有遗留未解答的问题，必须再次提交用户解答，并针对解答内容进行补充开发和自测，直到没有任何不确定性。
- 产出 Version 交付包：全部文档、测试报告、部署说明、**完整的问题清单历史**。
- 为 Version 打上语义化版本号 tag（如 `v2.1.0`）。

### 4. 测试策略总结

| 测试类型 | 触发时机 | 范围 | 环境 | 负责人 |
|----------|----------|------|------|--------|
| 单元测试（全量） | 每个 Stage 结束后 | 所有单元测试用例 | 共享测试环境 | MainAgent |
| 系统测试（增量） | 每个 Stage 结束后 | 当前 Stage 新增/影响的功能 | 共享测试环境 | MainAgent |
| 系统测试（全量） | Version 所有 Stage 结束后 | 整个系统 | 共享测试环境 | MainAgent |

> **注意**：由于只有一个 MainAgent 负责完整测试，所有测试都在同一个共享环境中串行执行，避免环境冲突。

### 5. 异常处理与重规划

- **SubStage 失败重试**：  
  - 由 MainAgent 通知对应 SubAgent 重试（最多 3 次）。  
  - 重试前可要求 SubAgent 提供问题分析。  
- **3 次失败后**：  
  - MainAgent 暂停该 Stage。  
  - 分析失败原因：任务拆分不合理？依赖遗漏？SubAgent 能力不足？  
  - 重新划分该 Stage 的 SubStage（可能改变并行粒度、增加文档要求）。  
  - 如必要，重写整个 Stage 并重新分配。  
- **不允许 SubAgent 擅自修改任务边界**：所有任务调整必须经 MainAgent 确认。
- **不确定性导致的暂停**：开发中任何 SubAgent 遇到不确定性问题，必须立即通过 MainAgent 追加到问题清单并寻求用户解答；未解答前不得继续该子任务。

### 6. 文档与门禁要求

| 工件 | 产生时机 | 格式 | 包含内容 |
|------|----------|------|----------|
| **Version 问题清单** | Version 规划阶段开始前 | Markdown | 所有关于 Version 需求的不确定性，多轮问答记录 |
| **Stage 问题清单** | 每个 Stage 开始前 | Markdown | 该 Stage 特有的不确定性，开发中追加的问题，最终全部解答 |
| **Stage 设计文档** | Stage 开始前，由 MainAgent 编写并自行审核 | Markdown | SubStage 列表、每个 SubStage 的输入/输出、跨 SubStage 契约、接口定义、预期测试用例。**必须满足"零上下文可开发"标准（见下方 6.1 节）** |
| **Stage 完成报告** | Stage 测试通过后 | Markdown | 实际完成情况、单元测试结果、增量系统测试结果、已知问题（如有） |
| **Version 总结报告** | Version 全量测试后 | Markdown | 所有 Stage 汇总、全量系统测试结果、语义化版本号、变更日志、问题清单归档 |

#### 6.1 "零上下文可开发"标准

> **核心要求：任何文档——无论是 Stage 设计文档还是 SubStage 任务书——都必须让一个全新的 session（没有本次对话历史、没有任何口头约定）仅凭该文档就能理解目标并直接开始开发。**

这是文档质量的强制门禁。MainAgent 在自行审核 Stage 设计文档时，必须逐条验证以下检查项，全部通过后方可分发：

**必须包含的内容：**

1. **背景与目标**：用 1-3 句话说明本 Stage / SubStage 要解决什么问题，达到什么状态。不得使用"如之前讨论的"、"按照需求"等引用性描述。
2. **当前代码状态**：说明相关文件/模块现在的状态（尚不存在、已有骨架、需要修改哪些具体位置）。涉及具体文件时给出相对路径。
3. **完整的接口契约**：所有跨 SubStage 或跨模块的接口、数据结构、枚举值、常量，必须在文档中完整定义（不得以"参考已有代码"替代，除非附上文件路径和行号）。
4. **验收标准**：明确的完成判定条件，SubAgent 可以自行判断"我做完了"——例如：哪些测试必须通过、哪些接口必须可调用、哪些行为必须可观测。
5. **禁止事项**：明确列出本 SubStage 不应触碰的模块、文件或行为，避免越界。
6. **依赖前置条件**：如果本 SubStage 依赖另一个 SubStage 的产物，必须说明该产物的存放位置和格式，而不是依赖执行顺序的隐式假设。

**审核自检问题（MainAgent 发布前逐条回答）：**

- 一个从未参与本 Version 讨论的 Agent，读完文档后，是否知道要改哪些文件？
- 文档中是否有任何"如前所述"、"按照之前的约定"、"参考对话记录"等需要额外上下文才能理解的表述？（有则必须展开）
- 接口定义是否完整到可以直接写代码？还是需要先去翻其他文件才能确定字段名和类型？
- 验收标准是否可以由 SubAgent 独立验证，无需询问 MainAgent？

### 7. 适用范围与限制

- ✅ 本范式**专用于大规模 Version**（需求点数量 ≥ 5，或存在明显可并行的子任务，或代码依赖复杂）。  
- ✅ 对于**小规模 Version**（需求点少、依赖简单、预计单人一天内可完成），只需由 **单个 MainAgent 独自开发 + 测试**，不拆分 Stage/SubStage，不引入 SubAgent。**但即使小规模 Version，也必须遵守“问题清单”原则**：开发前澄清所有细节，开发中遇到不确定性追加问题清单并寻求用户解答，确认无误后再继续。  
- ⚠️ 要求 MainAgent 有较强的任务分解、依赖分析以及**跨 SubStage 契约设计**能力。  
- ⚠️ 如果代码耦合度过高，无法拆分为无依赖的 SubStage，则优先重构解耦，否则该范式效果会大打折扣。  
- ⚠️ SubAgent 之间零沟通，所有协作必须上升至 MainAgent 协调，这会增加 MainAgent 的设计负担，但避免阻塞和混乱。  
- ⚠️ **问题清单是强制门禁**：未关闭所有问题前，不允许进入实际代码开发阶段。
