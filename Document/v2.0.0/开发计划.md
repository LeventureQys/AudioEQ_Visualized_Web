### 2.2 错误抛出约定（B4 决策）

```js
throw new Error('Band index out of range: ' + idx, {
  cause: { code: ErrorCode.INDEX_OUT_OF_RANGE, index: idx },
});
```

任何模块抛错都必须用此 pattern，`cause.code` 引用 `ErrorCode`。

### 2.3 事件命名规范（B3 决策）

所有事件名小写短横线，挂在 `EqualizerModel`（内部）或 `<audio-eq>` Element（对外）：

| 事件名 | detail 字段 |
|--------|-----------|
| `band-changed` | `{ index, band }` |
| `band-added` | `{ index, band }` |
| `band-removed` | `{ index }` |
| `band-count-changed` | `{ newCount }` |
| `focused-band-changed` | `{ index }` (index 为 -1 表示无焦点) |
| `sample-rate-changed` | `{ rate }` |
| `lpf-changed` | `{ frequency, enabled }` |
| `hpf-changed` | `{ frequency, enabled }` |
| `model-reset` | `{}` |
| `band-dragged` | `{ index, frequency, gain }`（仅在拖拽过程中由 Element 转发） |

**输出**：[src/core/CoordinateMapper.js](src/core/CoordinateMapper.js)。

**验收**：
- 构造时预计算 `logFreqRatio = Math.log(freqMax/freqMin)` 加速
- 频率对数映射公式：`x = vp.x + vp.w * Math.log(freq/freqMin) / logFreqRatio`

### 4.1 SubStage 2A：ButterworthIIR

**输入**：[../basic_code/CalculateCore.cpp](../basic_code/CalculateCore.cpp) L288-602（5 个 makeCoeff 函数 + FreqResponse），本文档第 2.1 节，《总览.md》"v1.0.0 已知 bug 修复要求"。

**任务**：实现 [src/core/filter/ButterworthIIR.js](src/core/filter/ButterworthIIR.js)：

```js
// 5 个纯函数（无副作用，所有参数显式）
export function makePeakBiquad(freq, q, gainDb, sampleRate): {b0,b1,b2,a1,a2}
export function makeLowShelfBiquad(freq, q, gainDb, sampleRate): {...}
export function makeHighShelfBiquad(freq, q, gainDb, sampleRate): {...}
export function makeLowPassBiquad(freq, sampleRate, q = 1/Math.SQRT2): {...}
export function makeHighPassBiquad(freq, sampleRate, q = 1/Math.SQRT2): {...}

// 频率响应求值
export function evaluateBiquadGainDb(coeff, freq, sampleRate): number  // dB

// 按 FilterType 一次性求值（业务层用这个）
export function evaluateBandGainDb(band, freq, sampleRate): number

// Q 值范围
export function qRangeFor(filterType): { min, max }

// Nyquist clamp 工具（重要 bug 修复）
export function clampFreqBelowNyquist(freq, sampleRate): number
```

**v1.0.0 bug 修复要求（强制）**：
1. **Nyquist clamp**：每个 `make*Biquad` 入口都先 `freq = clampFreqBelowNyquist(freq, sampleRate)`，避免 `tan(π·f/fs)` 在 `f ≥ fs/2` 时爆炸。ε 取 `1e-3`。
2. **Butterworth Q 解锁**：LPF/HPF 新增 `q` 参数（默认 `1/Math.SQRT2`，即标准 Butterworth -3dB 截止）。
3. **qmax 统一**：Peak/LowPass/HighPass/BandPass 的 qmax 一律 `128.0`（不是 v1.0.0 的 128.8）。
4. **死代码不抄**：CalculateCore.cpp L705-710 两分支相同的代码不复制。

**Q 值范围**（替代 v1.0.0 的 `FilterAlgorithm::qRange`）：

```js
qRangeFor(FilterType.PEAK)        → { min: 0.4, max: 128.0 }
qRangeFor(FilterType.LOW_SHELF)   → { min: 0.4, max: 1.6 }
qRangeFor(FilterType.HIGH_SHELF)  → { min: 0.4, max: 1.6 }
qRangeFor(FilterType.LOW_PASS)    → { min: 0.4, max: 128.0 }
qRangeFor(FilterType.HIGH_PASS)   → { min: 0.4, max: 128.0 }
qRangeFor(FilterType.BAND_PASS)   → { min: 0.4, max: 128.0 }
```

**输出**：[src/core/filter/ButterworthIIR.js](src/core/filter/ButterworthIIR.js)。

**验收**（在 Stage 7A 单元测试中验证）：
- Peak 滤波器在 `freq == centerFreq` 时返回 `≈ gainDb`（误差 < 0.5 dB）
- LowPass 在 `freq == cutoff` 时返回 `≈ -3 dB`（Butterworth Q=1/√2 标定）
- 各 makeCoeff 在 `freq > sampleRate/2` 时返回值不为 NaN/Infinity（Nyquist clamp 生效）
- 与 CalculateCore.cpp 同输入参数对比，系数误差 < 1e-9（除已知 bug 修复处）

**禁止事项**：
- 不引用任何 DOM / EventTarget / Class state——保持完全纯函数
- 不实现 BandPass（v1.0.0 也未实现，留待 v2.x 扩展时再加）

```js
  setQRange(filterType, min, max): void     // 自动 clamp 所有匹配 band 的 Q
  qRange(filterType): { min, max }
}
```

**事件**：所有 mutator 方法在状态变更后通过 `this.dispatchEvent(new CustomEvent('xxx', { detail }))` 发出，事件名见第 2.3 节。

**Band index 策略**（E3 决策）：
- 内部用 `Map<number, Band>` 存储，不用数组
- `addBand`：从 0 开始扫描，找到第一个未占用的 index
- `removeBand`：从 map 删除，不重排其他 index
- `setFocusedBandIndex(-1)` 表示无焦点

**LPF/HPF 内部存储**：
- `m_lpf = { frequency: 20000, enabled: false }`、`m_hpf = { frequency: 20, enabled: false }`
- 不混入 band map

**输出**：[src/core/EqualizerModel.js](src/core/EqualizerModel.js)。

**验收**：
- 构造时自动添加 5 个 band，频率按 `freqMin * (freqMax/freqMin)^(i/(N-1))` 分布
- `bandAt(idx)` 返回的 band 是 deep clone（外部修改不影响内部）
- 删除 band[2] 后 `addBand(...)` 返回 2（复用）
- 设置 `setQRange(Peak, 0.4, 2.0)` 时，已存在的 Peak band q=5.0 应被 clamp 到 2.0，并发出 `band-changed` 事件
- 所有 setter 在值未实际变化时**不发事件**（避免无效回调）
- 单元测试覆盖率 ≥ 90%

**禁止事项**：不引用 DOM、不引用 Canvas、不引用 CurveEngine。

### 4.3 SubStage 2C：CurveEngine（同步版本）

**输入**：[../数据流设计文档.md](../数据流设计文档.md) §1.2、[../模块设计文档.md](../模块设计文档.md) §3，本文档第 2 节。

**任务**：实现 [src/core/CurveEngine.js](src/core/CurveEngine.js)（**主线程同步版本**，不引入 Worker——基线决策 4）：

```js
export class CurveEngine {
  constructor({ pointCount = 500, freqMin = 1, freqMax = 24000 } = {}) { ... }

  setPointCount(count): void
  setFreqRange(min, max): void

  // 同步计算总曲线
  computeTotalCurve(model): Array<{ freq, gainDb }>

  // 同步计算单 band 曲线（焦点态用）
  computeSingleBandCurve(bandIndex, model): Array<{ freq, gainDb }>

  // 工具：生成对数频率点（不与 model 绑定）
  static generateLogFrequencyPoints(freqMin, freqMax, pointCount): number[]
}
```

**算法核心**：
1. 生成对数频率点：`freq[i] = freqMin * Math.pow(freqMax/freqMin, i/(pointCount-1))`
2. 对每个 freq[i]，累加所有非 bypass band 的 `evaluateBandGainDb(band, freq, sampleRate)`
3. 叠加 LPF/HPF 的增益（若 enabled），LPF/HPF 用 `makeLowPassBiquad`/`makeHighPassBiquad`
4. 返回 `[{freq, gainDb}]` 数组（**不做坐标变换**——坐标变换在 Render 层做，保持 CurveEngine 纯计算）

**输出**：[src/core/CurveEngine.js](src/core/CurveEngine.js)。

**验收**：
- 单 Peak band gain=+6dB, freq=1kHz, q=1.0 时，总曲线在 1kHz 附近峰值 ≈ +6dB
- LPF 关闭时不影响曲线
- LPF freq=1kHz enabled 时，曲线在 1kHz 处 ≈ -3dB
- 性能：500 点 × 10 band 单次计算 < 5ms（chrome devtools 实测）

**禁止事项**：
- 不引入 Web Worker、不引入 setTimeout/setImmediate 异步
- 不做坐标变换（这是 Render 层职责）
- 不发事件（CurveEngine 是无状态计算工具，消费方主动调用）

**任务**：实现 [src/render/CanvasRenderer.js](src/render/CanvasRenderer.js) + [src/render/theme.js](src/render/theme.js)：

```js
export class CanvasRenderer {
  constructor(canvas: HTMLCanvasElement, model, mapper, theme = Theme) { ... }

  // 生命周期
  resize(cssWidth, cssHeight): void          // 应用 DPR 缩放
  destroy(): void                             // 移除监听 / cancelAnimationFrame

  // 脏标记 + rAF 调度（C5）
  markDirty(): void                           // 由 Model/外部触发，下一帧重绘
  renderNow(): void                           // 立即同步重绘（测试用）

  // 数据更新接口（业务层调用）
  setTotalCurve(points: Array<{freq, gainDb}>): void   // 触发 markDirty
  setSingleBandCurve(bandIndex, points): void
  clearSingleBandCurve(): void

  // 主题覆盖
  setTheme(partial): void
}
```

**DPR 适配（C1）**：
- 构造时 `dpr = window.devicePixelRatio || 1`
- `canvas.width = cssWidth * dpr`、`canvas.height = cssHeight * dpr`
- `ctx.scale(dpr, dpr)`（之后绘制用 CSS 像素坐标）
- 监听 `matchMedia('(resolution: ...dppx)').addEventListener('change', ...)` 自动适配

**dirty flag + rAF（C5）**：
- `markDirty()` 设 `this._dirty = true`，若无 pending rAF 则 `requestAnimationFrame(this._renderLoop)`
- `_renderLoop()` 若 dirty 则清屏 + 依次调用 layer.draw(ctx)
- 不 dirty 则跳过（节能）

**绘制顺序**：背景 → GridLayer → LabelLayer → CurveLayer → HandleLayer。

**输出**：[src/render/CanvasRenderer.js](src/render/CanvasRenderer.js) + [src/render/theme.js](src/render/theme.js)。

**验收**：
- 在测试 HTML 中显示纯背景色（其他 layer 暂未填充）
- DPR=2 屏幕下 canvas backing store 是 CSS 尺寸的 2 倍
- 触发 markDirty 后下一帧重绘，连续多次 markDirty 仍只重绘一次
- 静态时（无 markDirty）requestAnimationFrame 不再调度

**禁止事项**：3A 不画曲线 / 网格 / 文字 / 手柄。

## 附录 A：常见疑问预案

| 疑问 | 解答 |
|------|------|
| 为什么不用 React / Vue？ | 用户决策（B 类问题）：`<audio-eq>` 是跨框架通用组件，不应绑定到任一框架 |
| 为什么不用 TypeScript？ | 用户明确要求"纯 JS"；通过 JSDoc + 自动生成 `.d.ts` 兼顾 TS 用户 |
| 为什么不上 Web Worker？ | 用户决策（基线 4）：500 点 × 10 band 在主线程 < 1ms，Worker 收益小但复杂度大 |
| 为什么不用 Catmull-Rom？ | 用户选 C3:A——500 点本身已足够密，先做简单版，不达标再升级 |
| 为什么 LPF/HPF 不用 magic number？ | 用户选 E2:C——JS 字符串/独立 API 比 magic number 更清晰，避免误用 |
| 为什么不发自动 npm？ | 用户选 F3:B——v2.0.0 还在 alpha，自动发包风险高 |
| 为什么默认 -24~+12 不对称？ | 用户选 D2:B——与目标样式.png 一致 |
