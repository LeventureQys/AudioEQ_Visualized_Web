## 5. 已知坑位

### 5.1 buildAxisVBO 被 11A 和 11B 同时改

详见 §4.5 合并顺序。**11A 在末尾留 `// === 11B will append 0 dB line here ===` 注释行**，11B 在该注释行下方插入 0 dB 追加代码。这样 git merge 不冲突（11A 改函数主体、11B 只在末尾追加 4 行）。

### 5.2 0 dB 颜色被 axis UBO 覆盖

[设计文档 §5.2 方案 a](./设计文档.md#5-substage-11b网格步长与档位) 决定：0 dB 横线和 L 形 axis 共用 `m_axisColor` 绘制。`m_gridZeroColor`（α=0.9）与 `m_axisColor`（α=1.0）的差异肉眼几乎不可分辨，可以共用 axis UBO。

**不要**为 0 dB 单独搞 `m_zeroDbVBO` + 单独 UBO，会导致：
- 多一次 vkCmdBindVertexBuffers + 多一次 UBO 写
- 与 §3.2.2 颜色契约的"轴线和 0 dB 共一档亮度"的视觉决策不一致

### 5.3 Scissor 顺序错乱导致黑屏

Vulkan 的 `vkCmdSetScissor` 必须在 `vkCmdBindPipeline` 之后（同帧任意位置），且 scissor 宽高必须 ≥ 1。若 SimpleDemo 启动黑屏：

1. 用 RenderDoc 抓帧查 scissor rect 是否被设成 (0, 0, 0, 0)
2. 检查 `m_size.width() - MARGIN_LEFT - MARGIN_RIGHT` 是否为负（窗口缩到 < 56 px 时会）
3. 在 `setInnerScissor` 里 `std::max(0, …)` 已经保护，但当 max=0 时绘制会被裁光——这是预期行为（窗口太小本来就显示不下）

### 5.4 单元测试 TestCoordinateMapper 可能失败

[tests/test_coordinate_mapper.cpp](../../../tests/test_coordinate_mapper.cpp) 内若有：
```cpp
EXPECT_EQ(mapper.gainMin(), -48.0);
```
之类的硬编码，11D 改默认值后会失败。

**正确处理**：
1. SubAgent **不要**自行改测试断言
2. 立即把失败行号 + 期望值贴到 [问题清单.md §5](./问题清单.md#5-开发中追加的问题动态维护) 追加一个 S11-Q9：
   > S11-Q9（11D 提）：`test_coordinate_mapper.cpp:Lxx` 断言 `gainMin == -48`，因 11D 改默认值为 -18 而失败。用户决定：
   > A. 同步改测试断言为 -18
   > B. 在测试用 `setGainRange(-48,+48)` 显式设置后断言
   > C. 其它
3. MainAgent 提交给用户解答后再继续

### 5.5 Scissor 切换次数多导致性能下降

每帧 4 次 `vkCmdSetScissor` 对性能影响 < 1%（每次几十纳秒），不必担心。如果未来要进一步合并，可以把 grid 和 curve 都用内口 scissor + 一次绑定，但本 Stage 不必优化。

### 5.6 字体 atlas 缺少 "K" / "H" 字符

Stage9-D 修好了字体 atlas，覆盖 ASCII 32-127。`K / H / z` 都在内。**若**视觉验收发现频率标签部分字符缺失，先核对：
```cpp
GlyphInfo gi;
qDebug() << "K:" << m_fontAtlas.glyphInfo('K', &gi);
qDebug() << "H:" << m_fontAtlas.glyphInfo('H', &gi);
```
都应返回 true。

### 5.7 freqToX(20Hz) 落在 MARGIN_LEFT 左侧

[设计文档 §3.2.4](./设计文档.md#324-频率档位-11b11c-共用) 的频率档位与 `freqMin=20` 一致，因此 `freqToX(20) = MARGIN_LEFT`（数学上落在内口左边界）。`20Hz` 标签居中后会向左溢出至 `MARGIN_LEFT - textWidth/2 ≈ 18 px`，仍在窗口内但部分压在左边距上——**这是预期行为（A1 答案 A）**。若觉得太挤，可在不动 `freqToX` 公式的前提下把 `MARGIN_LEFT` 从 44 → 50，但需要：
1. 上报问题清单
2. 与 [设计文档 §3.2.1](./设计文档.md#321-边距常量) 同步

**禁止**自行调整。

### 5.8 增益标签数字基线对齐 vs 字符串右对齐

[设计文档 §6.2 11C-3](./设计文档.md#6-substage-11c文字布局) 的代码用的是"字符串整体右对齐"。乍看像是字符串右对齐而不是"数字个位列对齐"，但因为：
- `"12"` 字符串 = 数字
- `"6"` 字符串 = 数字
- `"-12"` 字符串 = `"-"` + 数字

整体右对齐 ⇒ `"12"` / `"6"` / `"-12"` 的 `2` `6` `2` 三个个位字符的右边界完全在同一 X 上。这就是"数字基线对齐"的视觉效果。**不需要**为对齐拆开"负号绘制 + 数字绘制"。

### 5.9 LpfHandle 文字 "LPF"

[目标样式.png](../目标样式.png) 最右端的椭圆里有 "LPF" 字。当前 [LpfHandle](../../../src/LpfHandle.h) 实现是否包含这个文字渲染？**本 Stage 不修**——LPF Handle 的内文渲染若缺失，属于另一个 Stage 范围（v1.1）。视觉验收清单不强制要求 LPF 字样。

## 10. 常见 Q&A

**Q1：能不能为了"美观"在 11C 顺手把字体 atlas 升级到 16 px？**
A：不能。`LABEL_FONT_SIZE_PX = 12.0f` 是 §3.2.5 锁定的，需要改字号请上报问题清单。

**Q2：颜色采样值和设计稿真实色值有少量偏差怎么办？**
A：A6 答案是"近似值即可"。肉眼对照目标图无明显违和即视为通过。若有具体不满，用户会在视觉验收阶段提，届时改一行常量即可。

**Q3：能不能把 `m_axisColor` 也通过 `AudioEQ::setAxisColor` 开放？**
A：不在本 Stage 范围。A6 决定**只开放 curve + background**，其它颜色硬编码。需要开放走下个版本。

**Q4：SDK 现有 `setGainRange` 接口在 gain 范围内 clamp 还是 reject 超界 band gain？**
A：取决于 [EqualizerModel](../../../src/EqualizerModel.cpp) 现有实现，本 Stage 不修改 Model 行为，**仅改默认值**。clamp/reject 由 Stage5 决定，本 Stage 沿用。

**Q5：视觉验收时发现 0 dB 横线和频率标签有 1-2 px 重叠怎么办？**
A：调整 `FREQ_LABEL_Y_OFFSET_PX`（§3.2.5）从 4 → 6 即可，属于 11C 范围内的微调，不需要上报问题清单。

**Q6：能不能在 11A 顺手把 axis 改成 2 px 宽？**
A：A5 + A6 决定 axis 与 0 dB 共一档（`α=1.0`），1 px 即可。要加粗需上报问题清单。
