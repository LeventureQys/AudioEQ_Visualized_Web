## 2. 阻塞性问题（开发前必须关闭）

### A1 ｜频率轴的左侧空隙怎么处理？

**问题描述**：目标图第一档是 `20Hz`，距离左边缘有一段空隙（看起来 ≈ `MARGIN_LEFT + 20px`）。这是因为目标图：
- 用对数刻度（和当前一致）
- `freqMin = 20Hz`（确认沿用）
- **`20Hz` 标签的中心**在 `freqToX(20) = MARGIN_LEFT` 处，但文字本身往左/右各延伸半个字宽，导致视觉上标签紧贴左边缘

需要确认两件事：

| 选项 | 含义 |
|---|---|
| **A** | `20Hz` 标签**居中对齐到 `freqToX(20)`**，跟其它频率标签同处理，自然形成"标签的一半在左边距、一半在内口"的视觉。`MARGIN_LEFT` 至少要 `20px` 防止溢出窗口左边缘 |
| **B** | `20Hz` 标签**左边界对齐到 `freqToX(20)`**（即整个文字向右），左侧不留空隙，但这会让最左档与其它档对齐方式不一致 |
| **C** | `20Hz` 标签**右边界对齐到 `freqToX(20) - 4px`**（即标签完全在左边距内），第一档刻度仍在 `freqToX(20)` 处 |

> **推荐**：A — 视觉一致性。目标图就是 A。

**回复**
按照目标图来就好，A

### A2 ｜响度显示范围与 SDK 参数上限的关系？

**问题描述**：
- SDK 的 `EqualizerModel::gainMin/Max` 默认 `±48 dB`，用户可以通过 `AudioEQ::setGainRange(-24, +24)` 等接口修改，这是**编辑允许的范围**。
- 目标图显示的 **网格范围**是 `-18 ~ +12 dB`（不对称），这是**坐标轴可视范围**。
- 当前 `CoordinateMapper::m_gainMin/Max` 既被用于"映射"也被用于"网格生成"——两者绑定。

如果按目标图，需要把这两件事**解耦**：

| 选项 | 含义 |
|---|---|
| **A** | 把 SDK 默认 `gainRange` 直接改成 `(-18, +12)`，所有 band gain 编辑都限制在 `-18 ~ +12`。**简单但破坏向后兼容**：之前用户用 ±48 的项目会被 clamp |
| **B** | **保留** SDK 默认 `gainRange = ±48`，但**新增**"显示范围"概念（如 `CoordinateMapper::setViewGainRange(-18, +12)`），坐标轴 / 网格只画显示范围，但 band gain 可拖到超出显示范围（视觉上被 scissor 截掉，拖回则恢复）。**复杂但不破坏接口** |
| **C** | 折中：SDK 默认 `gainRange` 改为对称 `±18`（容纳目标图的下限 -18），显示范围 = 编辑范围对称，不引入"双范围"概念 |
| **D** | 折中：SDK 默认 `gainRange` 改为对称 `±24`（再大一点缓冲），网格步长仍 6 dB，目标图的不对称 (-18, +12) 视为"这张图刚好没显示到 -24"的偶然情况 |

> **推荐**：C 或 D — 引入"双范围"语义复杂度高、本 Stage 不值。**优先 C**（恰好包住目标图）。若你倾向"我希望编辑也只能在 ±12"，则选 A。

**请回答**：A / B / C / D / 其他

**回复**
这个可视范围是可以变换的，不只是视觉上，这个目标样式只是展示了其中一种情况，显示范围等于编辑范围，C，这个范围也是可以变动的，不要做两种范围

### A3 ｜响度轴是否完全去掉 dB 单位？

**问题描述**：目标图响度轴标签 `12 / 6 / 0 / -6 / -12 / -18`，**整列都没有 dB**。但频率轴每个标签都带 Hz/KHz。这两个不对称——属于设计取舍：

| 选项 | 含义 |
|---|---|
| **A** | 严格照搬目标图：响度全列**不写** `dB`，频率每档**带** `Hz/KHz` |
| **B** | 响度也带：`12dB / 0dB / -6dB`，更明确（不照搬，但符合 v1 用户原始诉求 "数轴要标记单位"） |
| **C** | 响度只在 **顶部端标**写一次 `dB`（轴最上端单独画 "dB" 二字），其它行只写数字 |

> **推荐**：A — 照搬目标图。响度标签密集（每 6 dB 一格 → 6 行），重复 `dB` 会很挤；且 `dB` 是惯例单位，省略不会引起歧义。

**请回答**：A / B / C / 其他
A吧
