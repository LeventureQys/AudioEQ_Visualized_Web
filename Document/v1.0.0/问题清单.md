### Q11. Vulkan 加载方式：volk vs 系统 SDK loader
**问题**：`thirdparty` 内置了 **volk**（Vulkan 函数指针动态加载器）和 **Vulkan-Headers**。这强烈暗示项目想**脱离系统 Vulkan SDK**，用内置 headers + volk 在运行时动态加载 Vulkan。但原设计文档（Stage2-2C 等）写的是 `find_package(Vulkan)` + 直接链接 `Vulkan::Vulkan`。两者不能混用。请确认：
- [ ] A. 使用 volk + 内置 vulkan-headers，完全不依赖系统 Vulkan SDK（所有 `#include <vulkan/vulkan.h>` 改为 volk 方式，启动时 `volkInitialize()`）
- [ ] B. 使用系统 Vulkan SDK 的 `find_package(Vulkan)`，volk 仅作为可选优化（或不用）
- [ ] C. 其他（请说明）

**背景**：这是全局架构决策，影响 VulkanContext、所有 9 个 Vulkan 模块的 include 和初始化代码。**必须在 Stage2 开始前确认**。

**MainAgent 推荐**：A。既然已内置 volk + headers，用 volk 可以最大化跨平台一致性（尤其 macOS），且不强制开发者安装 Vulkan SDK。

**回复**
A吧，目前先这么做

### Q12. volk 与 Qt QVulkanInstance 的协调
**问题**：若 Q11 选 A（用 volk），则 volk 的函数指针需要从一个有效的 `VkInstance` 加载（`volkLoadInstance`）。而 Qt 的 `QVulkanInstance` 会自己创建并管理 `VkInstance` 及其 loader。两者需要协调：
- [ ] A. 用 volk 自己创建 `VkInstance`，然后用 `QVulkanInstance::setVkInstance()` 把已创建的 instance 交给 Qt 包装（Qt 支持 adopt 既有 instance）
- [ ] B. 用 `QVulkanInstance` 创建 instance，再用 `volkLoadInstance(qtInstance->vkInstance())` 让 volk 从中加载函数指针
- [ ] C. 由 SubAgent（Stage2-2C 实现者）自行选择鲁棒方案

**背景**：影响 Stage2-2C（VulkanContext）和 Stage4（Qt-Vulkan 集成）的初始化时序。这是最容易出 bug 的地方。

**MainAgent 推荐**：B。让 Qt 主导 instance 创建（与 Qt 的 surface/窗口系统集成最顺），volk 从 Qt 的 instance 加载函数。

**回复**
B

### Q14. macOS / MoltenVK 支持的具体程度（接续 Q1）
**问题**：Q1 确认需要支持 macOS。Vulkan 在 macOS 上需通过 MoltenVK（转译到 Metal），且必须启用 portability 扩展（`VK_KHR_portability_enumeration` / `VK_KHR_portability_subset`）。原文档「Vulkan 不可用不降级」的决策在 macOS 上需要细化：
- [ ] A. macOS 上把 MoltenVK 也内置到 thirdparty（随库分发），开箱即用
- [ ] B. macOS 要求用户自行安装 Vulkan SDK（含 MoltenVK），CMake 查找
- [ ] C. v1.0.0 暂时只保证 Windows + Linux 真机验证，macOS 仅保证「能编译通过」，运行时验证推迟到后续版本

**背景**：影响 Stage2-2C（VulkanContext 需启用 portability 扩展/flag）、CMake 平台分支、以及 Stage8 兼容性测试的覆盖范围。MoltenVK 内置涉及二进制分发和许可。

**MainAgent 推荐**：C。优先保证 Windows+Linux 完整可用，macOS 保证编译通过 + 启用 portability flag，但真机测试不纳入 v1.0.0 门禁（写入已知限制）。否则 macOS 真机验证会显著拖慢 Version 进度。

**回复**
